# 闲鱼自动搜索项目开发复盘：从 Next.js 到飞书卡片通知

这是一篇关于 `xianyu-auto-search` 的开发复盘。这个项目的目标很明确: 用 Next.js 16 + Playwright 做一个可配置的闲鱼监控器，依托 Supabase 存储配置与去重记录，再把新增商品推送到飞书，形成一个真正闭环的自动化工作流。

## 一、项目背景与目标

最开始的问题不是“要不要做一个监控器”，而是“怎么把它做成一个能长期用的工具”。我希望它不是一次性的脚本，而是一个可以持续配置、持续运行、持续排错的系统。

项目的核心目标有三个。

第一，允许我在网页里直接配置搜索关键词、价格范围、发布时间范围、排序方式和通知地址，不需要改代码。

第二，监控逻辑要能稳定运行。不是简单地打开页面截图，而是要真的在闲鱼搜索页里完成筛选，再提取结果。

第三，通知要能闭环。发现新商品后，不只是打印日志，而是要准确发到飞书，并且支持去重，避免重复推送。

## 二、技术栈与架构

这个项目采用的是 Next.js 16 App Router，React 19，TypeScript 5，UI 组件基于 shadcn/ui，样式使用 Tailwind CSS 4。数据层使用 Supabase，爬虫层使用 Playwright，定时任务由 `node-cron` 驱动。

整体架构可以拆成四层。

第一层是前端配置页。它负责创建、编辑和触发监控任务，同时展示日志。

第二层是服务端 API。`/api/configs` 负责配置增删改查，`/api/trigger/[id]` 负责手动触发扫描。

第三层是监控调度与爬虫。`src/server.ts` 启动 Next.js 的同时也会启动监控任务调度器，定时去执行抓取。

第四层是通知与存储。Supabase 保存配置和已发送商品，飞书负责接收通知。

## 三、关键开发过程

开发过程中最先遇到的是开发服务锁文件问题。Windows 下 `next dev` 的子进程没有完全退出时，`.next/dev/lock` 会残留，导致再次启动时报“无法获取 lock”。后来我把 `scripts/dev.mjs` 改成了带子进程树清理的方式，避免残留开发进程。

第二个问题出在数据层。项目虽然已经接入 Supabase，但新建项目后表并不会自动存在。我根据 Drizzle schema 手动整理出初始化 SQL，把 `monitor_configs`、`sent_products` 和 `health_check` 建了出来，并补齐了浏览器调试相关字段。

第三个问题是爬虫提取。最开始的版本只是用一个通用选择器去找商品卡片，结果经常出现“找到了卡片，但提取不到商品”的情况。后来我根据真实页面结构改成了更稳的策略：先在闲鱼搜索页里真正点击筛选项，再从 `.feeds-item-wrap--rGdH_KoF` 等节点提取商品信息。

第四个问题是本地二次过滤。页面里已经通过闲鱼自带筛选拿到结果了，但我一开始还在本地继续做价格过滤，结果把 `2.88万` 这类值误判成了 `2.88 元`，把本来符合条件的结果又过滤掉了。最后我直接去掉了本地价格过滤，把它完全交给闲鱼页面本身处理。

## 四、重要实现点

### 4.1 Supabase

Supabase 主要承担两件事：保存监控配置和保存已发送商品。前者让网页配置变成可持久化数据，后者让去重真正有依据。

数据库表设计也比较直接。

- `monitor_configs` 保存关键词、价格区间、排序方式、Cookie、Webhook 地址和浏览器行为
- `sent_products` 用于去重，防止重复发送
- `health_check` 是预留的健康检查表

### 4.2 Next.js 自定义服务端

`src/server.ts` 不是普通的 Next.js 默认启动方式，而是自己包了一层 HTTP 服务。这样做的好处是，Next.js 页面和监控调度器可以在同一个进程里启动，前端和后端状态也更容易统一。

### 4.3 Playwright 与闲鱼筛选

爬虫最关键的改动不是“开浏览器”，而是“用浏览器真实执行筛选动作”。

现在搜索流程是先访问闲鱼搜索页，再尝试点击“新发布”“价格”“个人闲置”等筛选项，同时支持填写价格区间。抓到的页面截图和 HTML 也会保存到 `.next/debug/xianyu/`，方便回看真实页面。

### 4.4 飞书通知

通知链路后来也做了比较大的改造。最初是普通 JSON，后面改成飞书机器人更适配的卡片消息，并且按批次发送，避免一次塞太多内容。

现在的策略是每批 6 条商品，分批发到飞书。每一批都会记录请求体、HTTP 响应和飞书业务码，出问题时能直接定位。

## 五、踩坑与修复

这个项目里最典型的坑有三个。

第一，登录态问题。虽然浏览器打开后看起来是登录状态，但如果不用持久化用户目录，Playwright 的上下文并不会自动继承本机 Chrome 或 Edge 的登录信息。这个问题后来通过增加浏览器通道和用户目录配置来缓解。

第二，筛选生效问题。页面上看到了筛选控件，不代表代码真的操作到了控件本身。最后是通过 DOM 交互，而不是 URL 拼接，才让筛选真正生效。

第三，通知展示问题。飞书消息“成功发送”不代表内容完整送达，后来我把消息改成卡片分批发送后，信息才真正完整。

## 六、最终结果

现在这套流程已经比较完整了。

网页上可以直接创建监控任务，Supabase 保存配置和去重记录，服务端定时执行扫描，Playwright 在真实页面里做筛选，最后把新商品通过飞书卡片分批推送出去。

从结果上看，这个项目已经不是单纯的爬虫脚本，而是一个可配置、可调试、可持续运行的监控系统。

## 七、后续优化方向

后面还可以继续优化三个方向。

第一，把飞书卡片做得更像一个真正的通知面板，比如加入封面、标签和更清晰的摘要。

第二，把爬虫的页面操作再做得更稳一些，尤其是对闲鱼前端变化的适配。

第三，把日志和调试产物继续产品化，例如在前端直接展示最近一次截图、最近一次响应体和最近一次卡片发送结果。

## 八、总结

这次项目最重要的收获不是“把功能做出来”，而是把整个链路打通：配置、抓取、筛选、去重、通知、日志、调试，全部都有了可追踪的落点。

当一个自动化项目真正跑起来之后，最值钱的不是某一段代码，而是整条闭环是否能稳定运转。这个项目现在已经走到了这一步。
